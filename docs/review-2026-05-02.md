# Egypt Deals — Review (2026-05-02)

## 1. Summary (TL;DR)

The app is in solid shape for a personal PWA. Frontend builds clean, scraper pulled 184 deals (30 Amazon + 154 Noon) without errors, push delivery works end-to-end, the deal-detail price assessment is genuinely useful, and the live site at https://egypt-deals.vercel.app renders well at iPhone width. This pass landed 7 fixes (one of them caught only by smoke-test on the live deploy) plus an a11y/copy polish bundle. The most visible improvements are the skeleton grid + stale-data banner on first load, properly-clearing dead push subscriptions, and a real "Newest" sort. Biggest open question: the `priceCount >= 1` ATL threshold is intentionally loose but means most "ATL" claims early on are weak — needs a product call.

## 2. Bugs found and fixed

- **"Newest" sort was a no-op** — `scraper/index.js:62` sets `scrapedAt = Date.now()` for every deal in the run, so sorting on it never reordered anything. Switched to `firstSeenTs` (per-product earliest record) — commit `9d1a32b`. Users would have noticed the Newest sort showed the same order as whatever sort they had before.
- **Service-worker offline fallback was dead** — `public/sw.js` intercepted navigation with `caches.match('/')` but nothing ever populated the cache, so offline navigation rejected silently with `undefined`. Removed the handler so the browser uses its native cache + offline page. Commit `cb4ccf3`. Users on poor connectivity would have seen the PWA fail to load entirely instead of getting a useful error.
- **SW default icon path mismatch** — pushed payload without an explicit `icon` fell back to the pre-cache-bust `/icons/icon-192.png`. Bumped to `-v3.png`. Commit `cb4ccf3`. Notification icon could have been stale.
- **Stale push subscription never cleared** — `scraper/push.js` logged on 404/410 but kept the dead endpoint in Upstash, so every subsequent scrape retried it. Now we `redis.del('push_subscription')` on those status codes. Commit `b8f91b7`. After uninstalling the PWA the user would see persistent push errors in the scraper logs and re-subscribing cleanly required manual Redis intervention.
- **Single-deal notification icon path inconsistency** — single-deal pushes used `/icons/icon-192.png`, digest used `-v3.png`. Unified to `-v3.png`. Commit `b8f91b7`.
- **Header showed "Loading" forever after empty fetch** — `/api/deals` returned `updatedAt: null` but the header bound `snapshot?.updatedAt ? timeAgo(...) : 'Loading'`. API now returns `0`, header now shows "No data yet". Commit `f2c8cdb`. First-ever cold start (before any scrape) would have looked broken.
- **Amazon deep-link double-opened web tab** — `lib/deep-link.ts` opened the web fallback unconditionally 500ms after the `amazon://` scheme. Now checks `document.hidden` so the fallback is suppressed when the app handled it. Commit `526309d`. Users with the Amazon app installed would have seen both the app open AND a stale browser tab linger.
- **Settings "1 minutes ago" pluralization** — caught on live smoke test. Commit `1deb773`.

## 3. Bugs found but NOT fixed

- **`isAllTimeLow` returns true at `priceCount >= 1`** — `scraper/atl.js:1`. Comment from the prior session calls this intentional, but it means every brand-new product becomes "ATL" on first sight, which makes the green "ATL (our data)" badge meaningless until ~5 scrapes have accumulated. Out of scope for this pass — needs a product decision (see §5 SHOULD).
- **"Near the lowest seen — within EGP 0 (0-day window)"** — verified live on a Noon mouse deal. When `min === max === current` (a freshly-tracked product where every sample has been the same price), `assessPrice()` lands in the `< 0.25` bucket and prints awkward zeros. Needs a copy/logic decision: should this case map to "Just started tracking" (which currently only triggers for `priceCount < 3 && days < 1`)? Likely yes, but it changes verdict semantics — flagging for the user to call.
- **Smart-verdict "Price is volatile" can mislead** — `scraper/atl.js:28` falls back to "volatile" whenever `recentPrices` aren't strictly monotonic, even for noise of 1 EGP across 4 samples. The detail page now hides this block when `priceCount < 3`, but for products with 3+ flat samples it still prints "volatile". Worth tightening with a real variance threshold but not a critical bug.
- **`scrapedAt` field is uniform per run** — see §6.
- **No `.gitignore` for `tsconfig.tsbuildinfo` and `scraper/data/`** — both got tracked accidentally during this session. Worth adding but not user-visible.

## 4. Polish applied (single commit `e7ef7eb`)

- Comma-separated prices everywhere: `EGP 12,999` instead of `EGP 12999`, via a shared `fmtEGP` helper in `DealCard.tsx` and `app/deal/[id]/page.tsx`.
- `aria-pressed` reflects active state on filter chips; `role="tablist"` on each row; `aria-label`s ("Filter by store", "Filter by category").
- `focus-visible:ring-2 focus-visible:ring-blue-500` on deal cards and filter chips so keyboard users see focus.
- DealCard `aria-label` includes name + price + discount so screen readers don't read raw "EGP 12999" as a single number.
- Settings push toggle: when `Notification.permission === 'denied'` the toggle is disabled with helper copy "Blocked — re-enable in iOS Settings → Notifications → Deals". Previously it just `alert()`ed something useless on tap.
- Deal detail: hide the "Smart verdict" block when `priceCount < 3` (the price-assessment block above already covers the not-enough-data case, and the smart verdict at <3 points is generic).
- Skeleton grid (6 pulsing cards) on first load instead of a blank page with "Loading" in the header.
- Stale-data amber banner when `Date.now() - updatedAt > 3h`.

## 5. What's missing — recommended next

### MUST

- **Product call: tighten ATL threshold (S)** — should `isAllTimeLow` require `priceCount >= 5` (or some confidence floor) before stamping the green "ATL (our data)" badge? Today every product is "ATL" on first scrape. **Question for Ahmed:** OK to gate ATL on `priceCount >= 5`? It will cut the feed empty for ~5 hours after a fresh DB reset but make every shown deal trustworthy.
- **Edge-case verdict copy: handle "all samples identical" (S)** — `assessPrice()` should detect `range === 0 && priceCount >= 3` and print something honest like "Price has been flat at EGP X for {days}d". The current "Within EGP 0 of the all-time low (0-day window)" reads as broken UX. User-visible win: the most-clicked detail view stops looking janky.

### SHOULD

- **Empty-image fallback (S)** — `DealCard` renders the 📦 emoji when `imageUrl` is empty (e.g. the men's socks card on the live site). Either skip such items in the snapshot, or render a more graceful generic image. Win: feed feels finished, not buggy.
- **Manual scrape spinner accuracy (M)** — `app/page.tsx` polls `/api/deals` 18× over 90s waiting for `updatedAt > before`. If the user's PC is off, this spins for 90s then silently gives up with no feedback. Add a "couldn't reach scraper — your PC may be off" toast on timeout. Win: trust.
- **Settings: scraper health card (S)** — show actual `last_scrape_ts` (we already store it!) plus next expected run estimate. Currently "Last updated" piggybacks on the snapshot timestamp. Win: clearer mental model when push fails.
- **Wishlist / saved deals (M)** — let the user star a product and get notified specifically when *that* product hits a new low, even if it's not in the top-50 ATL feed. Win: the feature people actually use price trackers for.
- **iOS PWA test pass (S)** — verified at desktop-emulated iPhone width via Chrome MCP, but Ahmed should install on iPhone Home Screen and confirm the safe-area inset, the touch refresh, the swipe-back gesture, and most importantly that a delivered push tapped from lock screen actually navigates to `/deal/<id>`.
- **Search debouncing + history (S)** — search currently re-filters every keystroke (cheap, but no analytics on what was searched). Persist last 5 searches to localStorage and surface as chips. Win: faster repeat lookups.
- **Tighten "Smart verdict" volatility heuristic (S)** — see §3. Compute stdev relative to mean; only call "volatile" when stdev/mean > ~5%.

### COULD

- **Per-deal "watch this" → personal price target alert (M)** — extension of wishlist; user enters a target price, gets a push when it crosses.
- **Scraper failure visibility (S)** — currently a 0-deals scrape result silently overwrites the snapshot with an empty deals array. Either skip the snapshot write on `allDeals.length === 0` or show "Last scrape failed" in settings.
- **Batch-screenshot share image (S)** — long-press a deal → generate a price-history PNG to share to friends. Pure nice-to-have.
- **Categorize "general" / "other" buckets (S)** — current classifier sends ~30% of products to `other`. Look at what's landing there for a week and tighten rules. Win: filters feel exhaustive.
- **PWA install prompt (S)** — first visit on iOS Safari, show a one-time tip explaining the Share → Add to Home Screen flow. Win: more push subscribers.
- **Dark/light theme toggle (M)** — the app is fully dark; iPhone OLED users will love it but a light mode for outdoors visibility wouldn't hurt.

## 6. Architecture notes

- **`scrapedAt` is uniform across a scrape run.** It's set to `Date.now()` for every deal in `scraper/index.js:62` so it's effectively just a "snapshot timestamp" duplicated onto every record. The frontend type `Deal.scrapedAt` invites confusion. Either drop it from the per-deal payload (use `DealsSnapshot.updatedAt` instead) or genuinely set it from the SQLite row's `MAX(scrapedAt)`.
- **`getSecondLowestPrice` returns the *lowest* price when only one distinct value exists** (`db.js:81`), not null. Consumers in `index.js` treat it as "previous low," which then equals current price, which makes the verdict ratio `1.0` → falls into the "matches previous all-time low" branch. This is OK in practice but a subtle naming lie.
- **`upsertProduct` writes the `imageUrl` it just scraped on every run.** If a category page shows a different thumbnail size for the same product on a later run, the URL flickers. Probably fine; flagging for awareness.
- **No idempotency on `setDealsSnapshot`.** Two simultaneous scrapes (poller + manual + scheduled) could race and one's snapshot would clobber the other. Today only one scraper runs at a time (the poller has `inFlight` guard) but if `runScrape()` is ever invoked from two scripts the snapshot becomes whichever finishes second.
- **`@upstash/redis` is duplicated in both `lib/upstash.ts` (frontend) and `scraper/upstash.js`** — that's fine for keeping client-only and node-only code separate, but the new `clearDeadSubscription` in `scraper/push.js` instantiates a *third* Redis client. Could refactor to share a single accessor inside `scraper/`.
- **Type `Deal` lives only in `lib/types.ts`** — scraper-built objects aren't typed. If scraper ever drops a field the frontend breaks at runtime. A shared `.d.ts` or runtime validator (zod) would help but is overkill for a personal app.
- **Service worker has no versioning or update strategy.** It calls `skipWaiting()` + `clients.claim()` on every install, which is correct for a PWA where you want freshness over freshness-control. Just noting.
- **`tsconfig.tsbuildinfo`, `scraper/data/`, and the captured `amazon-debug.html` got committed during this session.** Should be in `.gitignore`. Out of scope for this pass.
